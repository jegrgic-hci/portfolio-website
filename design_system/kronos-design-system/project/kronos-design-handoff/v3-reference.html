Kronos v3

The Cerulean direction is canonical. The "pick A or B" picker is retired. v2.5's three experimental upgrades — 11-step neutrals, elevation tokens, density modes — are no longer experiments; they're the foundation. The namespace moves to --k3-* so downstream consumers know they've moved off the picker.

01

What v3 inherits from v2.5

No regressions — every v2.5 token resolves under the new namespace. The four additions below are now load-bearing, not experimental.

Foundation 11-step neutral scale

Cool 1° tint. --k3-n-50 through --k3-n-950. Aliases --k3-fg-1..4 resolve into the scale.

Foundation Elevation, 2 tokens

--k3-elev-1 for popovers/toasts. --k3-elev-2 for modals. Cards and buttons stay flat.

Foundation Density modes

Default = cozy. data-density="compact" tightens rows, lists, inputs. Buttons and cards opt out.

Foundation Named motion

Quick 80ms / base 150ms / emphasis 240ms / enter 320ms. Single easing curve. prefers-reduced-motion respected.

04

v3 charter — open for this sprint

Four candidate workstreams. v3 doesn't need all four to ship — but we should pick the ones we commit to before building. Each card lists the questions I'd want answered before starting.

A · Surface

Dark mode

Inverse surface scale + on-dark text alphas exist already (sidebar uses them). A full dark mode means committing to dark-equivalent values for every component surface, not just chrome.

  • Pure inversion, or a separate cool-grey-on-near-black palette?
  • Cerulean 500 vs 400 on dark — which reads more "brand"?
  • Warm-rust complement keeps its hue, or shifts cooler on dark?
  • Auto via prefers-color-scheme, or explicit class toggle?
Effort · Medium · ~3 days
B · Data

Data-viz palette

Charts in the dashboard currently improvise — they reach for accent, complement, then neutrals. A real series palette would let us label up to 8 categories without aesthetic drift.

  • Categorical (qualitative) only, or also sequential + diverging?
  • Anchor the first three series to accent / complement / neutral?
  • Color-blind safe (deuteranopia + protanopia) — required or nice-to-have?
  • Do we ship pattern fills for B&W print?
Effort · Small · ~1.5 days
C · Motion

Motion maturity

v2.5 added named durations. v3 could codify the where — which components move, which never do, and how multi-element transitions coordinate (stagger? cascade? respect reduced-motion to what degree?).

  • Stagger token (e.g. --k3-stagger-base: 40ms) — yes/no?
  • Allowed transforms — still none, or add a single 2px lift?
  • Page transitions — view-transitions API, or stay static?
Effort · Small · ~1 day · low risk
D · Forms

Form patterns

Single inputs are documented; multi-field forms are not. Layout, error summary, inline validation timing, helper-text positioning, required markers — none of it is standardised. Construction product needs this immediately.

  • Label position — top, left, or context-dependent?
  • Error summary at top, inline per field, or both?
  • Required marker — asterisk, "(required)", or "(optional)" inverse?
  • Save behavior — explicit submit, autosave, or per-field commit?
Effort · Medium · ~2.5 days
